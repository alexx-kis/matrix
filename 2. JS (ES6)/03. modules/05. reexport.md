# reexport

- это когда модуль импортирует что-то и сразу экспортирует дальше
- то есть модуль становится посредником

### без reexport

```js
// mathUtils.js

export function sumNumbers(firstValue, secondValue) {
  return firstValue + secondValue;
}

// textUtils.js

export function makeUppercase(textValue) {
  return textValue.toUpperCase();
}

// теперь в другом файле нужно импортировать всё отдельно:

import { sumNumbers } from './mathUtils.js';
import { makeUppercase } from './textUtils.js';
```

## что делает reexport

- можно создать один центральный файл:

```js
// utils/index.js

export { sumNumbers } from './mathUtils.js';
export { makeUppercase } from './textUtils.js';
```

- теперь импорт выглядит так:

```js
import { sumNumbers, makeUppercase } from './utils/index.js';
```

## зачем это нужно?

1. удобный единый вход (single entry point)

- вместо:

```js
import { alphaTool } from './tools/alphaTool.js';
import { betaTool } from './tools/betaTool.js';
import { gammaTool } from './tools/gammaTool.js';
```

- можно:

```js
import { alphaTool, betaTool, gammaTool } from './tools/index.js';
```

- это делает API проекта чище

2. скрытие внутренней структуры проекта

- например:

```
src/
  helpers/
    array/
      arrayHelpers.js
```

- без reexport:

```js
import { removeDuplicates } from './helpers/array/arrayHelpers.js';
```

- с reexport:

```js
import { removeDuplicates } from './helpers/index.js';
```

- теперь можно менять внутренние папки проекта, не ломая импорты в остальных файлах

3. удобно для библиотек

- очень часто библиотеки делают главный файл `index.js`, который собирает все экспорты

- например:

```js
export { ModalWindow } from './components/ModalWindow.js';
export { DropdownMenu } from './components/DropdownMenu.js';
export { TabsPanel } from './components/TabsPanel.js';
```

- тогда пользователь библиотеки пишет:

```js
import { ModalWindow, TabsPanel } from 'my-library';
```

- а не:

```js
import { ModalWindow } from 'my-library/components/ModalWindow.js';
```

## виды reexport

1. reexport named export

```js
export { calculatePrice } from './priceUtils.js';
```

2. reexport с rename (`as`)

```js
export { calculatePrice as getPrice } from './priceUtils.js';
```

- теперь наружу экспортируется имя `getPrice`

3. reexport всего через `*`

```js
export * from './dateUtils.js';
```

- это значит "экспортируй все named exports из этого файла"

### важный момент про default export

- default export нельзя просто так реэкспортнуть через `*`
- это не сработает:

```js
export * from './logger.js';
```

- если внутри:

```js
export default function loggerFunction() {}
```

- потому что `*` экспортирует только named exports

- для default нужен отдельный синтаксис

```js
export { default } from './logger.js';
```

- или:

```js
export { default as loggerFunction } from './logger.js';
```

## очень частый паттерн - `index.js`

- например:

```
components/
  Button.js
  Input.js
  Modal.js
  index.js
```

- теперь:

```js
import { ButtonComponent, InputComponent } from './components';
```

## плюсы reexport:

1. чище импорты
2. удобный public API
3. можно менять внутреннюю структуру проекта
4. меньше длинных путей
5. удобно для библиотек

## минусы reexport

1. иногда сложнее понять источник

```js
import { createRequest } from './utils';
```

- где именно находится функция? нужно идти в index.js

2. возможны конфликты имён

```js
export * from './userUtils.js';
export * from './adminUtils.js';
```

- если в обоих файлах есть

```js
export function formatData() {}
```

- будет конфликт

## главное запомнить

- reexport нужен чтобы
  - собирать экспорты в одном месте
  - делать удобный API
  - скрывать структуру проекта
  - упрощать импорты
