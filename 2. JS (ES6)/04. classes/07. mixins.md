# mixins

- это способ добавлять классам дополнительные возможности без inheritance

## зачем нужны mixins?

- пусть есть классы:
  - User
  - Button
  - Modal
  - VideoPlayer

- и всем нужен:

```js
show();
hide();
```

- через inheritance неудобно, потому что

```js
class User extends Something
class Button extends Something
class Modal extends Something
```

- но JS не поддерживает multiple inheritance, то есть так нельзя:

```js
class Modal extends UIComponent, AnimationComponent // ошибка
```

- поэтому используют mixin
- это объект или функция, которая добавляет методы

- самый простой mixin:

```js
const visibilityMixin = {
  show() {
    console.log('Showing element');
  },
  hide() {
    console.log('Hiding element');
  },
};
```

- добавим mixin классу

```js
class ModalWindow {
  //
}

Object.assign(ModalWindow.prototype, visibilityMixin);
```

- Object.assign() - копирует методы в prototype

- теперь методы доступны

```js
const settingsModal = new ModalWindow();

settingsModal.show();
settingsModal.hide();
```

- что произошло внутри?

```js
// было:

ModalWindow.prototype;

// стало:

ModalWindow.prototype.show;
ModalWindow.prototype.hide;
```

- то есть mixin просто добавляет методы

### важно понять

- mixin не создаёт inheritance chain, просто копируются методы
- поэтому mixins ≠ inheritance

### ещё пример

- mixin для логирования

```js
const loggerMixin = {
  log(messageText) {
    console.log(`[LOG]: ${messageText}`);
  },
};

class ApiService {
  //
}

Object.assign(ApiService.prototype, loggerMixin);

const backendApi = new ApiService();

backendApi.log('Request started');
```

## несколько mixins

- можно добавлять несколько mixins

```js
const eventMixin = {
  emit(eventName) {
    console.log(`Event: ${eventName}`);
  },
};

class ChatWidget {
  //
}

Object.assign(ChatWidget.prototype, loggerMixin, eventMixin);

// теперь объект имеет методы обоих mixins

const supportChat = new ChatWidget();

supportChat.log('Chat opened');

supportChat.emit('message');
```

## почему mixins популярны

- потому что inheritance быстро становится сложным

### проблема deep inheritance

```
AdminPanel
  ↓
DashboardPanel
  ↓
PageComponent
  ↓
BaseComponent
  ↓
UIElement
```

- такие цепочки:
  - сложно поддерживать
  - сложно дебажить
  - сложно расширять

### mixins дают composition

- вместо "is a"
- используют "hasAbilities"

- например, не Modal is AnimatedComponent, а Modal has animation behavior

- это ближе к composition pattern

## mixin через function

- иногда mixins делают как функции

```js
function addTimestampMixin(targetClass) {
  targetClass.prototype.showTimestamp = function () {
    console.log(Date.now());
  };
}

class NotificationPopup {
  //
}

addTimestampMixin(NotificationPopup);

// теперь:

const popupMessage = new NotificationPopup();

popupMessage.showTimestamp();
```

## продвинутый modern pattern

- часто mixins делают через class factories

```js
const animationMixin = (BaseClass) =>
  class extends BaseClass {
    animate() {
      console.log('Animating');
    }
  };

class UIBlock {
  //
}

class SidebarMenu extends animationMixin(UIBlock) {
  //
}

const sidebarInstance = new SidebarMenu();

sidebarInstance.animate();
```

- что происходит:

```js
animationMixin(UIBlock) // возвращает новый класс
class extends UIBlock // с дополнительными методами
```

- это очень популярный modern pattern
- особенно:
  - react ecosystem
  - decorators
  - framework internals
  - libraries

## конфликт имён

- очень важная проблема mixins

```js
const firstMixin = {
  show() {
    console.log('First');
  },
};

const secondMixin = {
  show() {
    console.log('Second');
  },
};

Object.assign(ModalWindow.prototype, firstMixin, secondMixin);
```

- метод второго mixin перезапишет первый, потому что Object.assign делает overwrite
- поэтому mixins требуют аккуратности

### важный момент

- mixin обычно не хранит state
- хороший mixin чаще всего
  - добавляет behavior
  - добавляет utility methods

- а не

```js
this.users = [];
this.balance = 100;
```

- потому что может возникнуть конфликт данных

## mixins vs inheritance

```
| Inheritance       | Mixins                      |
| ----------------- | --------------------------- |
| жесткая структура | гибкое добавление поведения |
| deep hierarchy    | composition                 |
| один parent       | можно много mixins          |
| strong relation   | reusable behavior           |
```

- в реальных проектах очень часто встречаются mixins для:
  - events
  - animations
  - logging
  - validation
  - serialization
  - draggable behavior
  - selectable behavior

- но сейчас composition обычно популярнее
- во многих современных проектах предпочитают:
  - composition
  - hooks
  - utilities
  - helper functions
- вместо сложных mixin systems

## самое главное понять

- mixin – это способ добавить классу поведение без inheritance
- обычно для копирования методов или создания нового класса поверх старого
