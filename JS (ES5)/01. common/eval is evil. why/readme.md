# "eval is evil". Why?

## 1. что такое `eval`

- `eval` - это встроенная функция, которая выполняет строку, как код

```js
const code = 'console.log(2+3)';
eval(code); // 5
```

## 2. проблемы с `eval`

### 1. безопасность

- если `eval` выполняет внешний или пользовательский ввод, это открывает дверь для атак типа `code injection`

```js
const userInput = "alert('hacked!')";
eval(userInput); // выведет alert — опасно!
```

- то есть кто-то может внедрить вредоносный код, если используется `eval` для пользовательских данных

### 2. производительность

- код, выполняемый через `eval` не оптимизируется движком JS (V8, SpiderMonkey)
  - движок не может заранее проанализировать такой код
  - из-за этого `eval` работает медленнее, чем обычный код

### 3. код в `eval` труднее читать и отлаживать

```js
eval('let x = 5; console.log(x)');
```

- стек ошибок будет показывать только `eval`, без конкретной строки
- инструменты типа линтеров или TypeScript не могут проверить такой код на ошибки

### 4. scope и переменные

- `eval` может менять локальное и глобальное окружение, что делает код непредсказуемым:

```js
let a = 1;
eval('a = 10');
console.log(a); // 10
```

## безопасные альтернативы

1. JSON.parse / JSON.stringify для работы с данными вместо `eval`
2. функции или объекты для динамического поведения
3. Function constructor (`new Function(...)`) - чуть безопаснее, но почти всегда можно обойтись без него

```js
// плохо
eval('2 + 3');

// хорошо
const result = 2 + 3;
```

## итог

`eval` считается "evil" потому что:
1. открывает двери для XSS и инъекций кода
2. замедляет работу кода (нет оптимизации)
3. создаёт сложности с отладкой и безопасностью переменных
4. почти всегда есть безопасные альтернативы
