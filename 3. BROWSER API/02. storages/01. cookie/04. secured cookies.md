# secured cookies

- когда cookie начали использовать для авторизации и хранения пользовательских данных, появилась проблема безопасности

- если хранить cookie без дополнительных настроек, ее могут:
  - украсть
  - подделать
  - отправить не туда, куда нужно
  - прочитать через JS

- поэтому у cookie появились специальные security attributes (флаги безопасности):

## флаг secure

- означает отправить cookie только по HTTPS

```http
Set-Cookie: sessionId=abc123; Secure
```

## флаг HttpOnly

- означает, что JS не может читать или изменять эту cookie

```http
Set-Cookie: sessionId=abc123; HttpOnly
```

- cookie с HttpOnly:
  - браузер все равно отправляет серверу
  - JS ее не видит
- это стандартный способ хранения session cookie

## флаг SameSite

- защищает от атак типа CSRF (Cross-Site Request Forgery)
- допустим, пользователь авторизован на сайте bank.com и имеет cookie sessionId=abc123
- затем пользователь открывает другой сайт evil-site.com, пытается отправить запрос на bank.com
- без ограничений браузер может автоматически приложить cookie банка к этому запросу - это потенциально опасно

### SameSite=Strict

- самый строгий вариант

```http
Set-Cookie: sessionId=abc123; SameSite=Strict
```

- cookie отправляется только если пользователь находится на самом сайте
- максимальная безопасность

### SameSite=Lax

- самый распространенный вариант

```http
Set-Cookie: sessionId=abc123; SameSite=Lax
```

- разрешает некоторый переходы между сайтами, но блокирует большинство опасных сценариев
- во многих браузерах это значение по умолчанию

### SameSite=None

```http
Set-Cookie: sessionId=abc123; SameSite=None
```

- разрешает отправлять cookie в кросс-доменных запросах

- но есть обязательное требование:

```http
Set-Cookie: sessionId=abc123;
             SameSite=None;
             Secure
```

- без `Secure` современные браузеры такую cookie не примут

## типичная безопасная cookie

- в продакшене часто можно увидеть:

```http
Set-Cookie: sessionId=abc123;
             Secure;
             HttpOnly;
             SameSite=Lax
```

- такая cookie:
  - отправляется только по HTTPS
  - недоступна JS
  - защищена от большинства CSRF-сценариев

## можно ли устанавливать флаги через JS?

- частично
- например:

```js
document.cookie = 'theme=dark; Secure'; // работает

document.cookie = 'sessionId=abc123; HttpOnly'; // не работает, браузер игнорирует HttpOnly
// потому что это бессмысленно
// JS создал cookie -> запретил себе ее читать
// поэтому HttpOnly может устанавливать только сервер через `Set-Cookie`
```

## что используют для авторизации в React/Next

- чаще всего:

```http
Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Lax

// или

Set-Cookie: accessToken=...; HttpOnly; Secure; SameSite=Lax
```

- потому что это намного безопаснее, чем хранить токены в LocalStorage
